## Setup performed

1. `mkdir -p /home/apps/deep-research-backend`, code synced via `rsync`
   (no git remote configured for this repo yet — see Known Limitations).
2. `git submodule add --depth 1 ... vendor/searxng` / `vendor/crawl4ai`,
   each checked out on a `local-patches` branch at the pinned commit above
   (ARCHITECTURE.md §13.2).
3. `.env` created from `.env.example`, with real generated values for
   `SEARXNG_SECRET` and `CRAWL4AI_API_TOKEN` (see below) —
   `python3 -c "import secrets; print(secrets.token_hex(32))"`.
4. `docker compose build` (all five services), `docker compose up -d`.
5. `docker compose exec api alembic upgrade head`.
6. One API key seeded directly (no admin endpoint exists yet — see Known
   Limitations):
   ```sql
   INSERT INTO api_keys (id, key_hash, label, rate_limit_per_minute)
   VALUES (gen_random_uuid(), '<sha256 of the raw key>', 'e2e-test-key', 60);
   ```
7. Host port later changed from the `8000` default to `10001`
   (`API_PORT=10001` in `.env`) to avoid colliding with other projects on
   this shared host — `docker-compose.yml`'s `api.ports` reads
   `${API_PORT:-8000}:8000`, container-internal port unchanged.

## Bugs found and fixed by deploying for real

Everything below was invisible in unit/contract tests (which mock the
provider boundary) and only surfaced by actually running the stack. Each
is fixed in code, tested, and documented — this is the point of an actual
deployment, not a gap to apologize for.

1. **SearXNG's build is two Dockerfiles with a cross-image dependency.**
   `container/dist.dockerfile`'s first stage is `FROM
   localhost/searxng/searxng:builder`, expecting that image pre-tagged
   locally — something plain `docker compose build` can't express.
   `docker/searxng/Dockerfile` merges both stages into one multi-stage
   build instead. (ARCHITECTURE.md §13.2)
2. **SearXNG's `searx/version_frozen.py` doesn't exist in a fresh shallow
   clone** — it's generated by `python searx/version.py freeze`, which the
   Dockerfile's `COPY` step needs to already exist. Run once after cloning
   the submodule, before the first build:
   ```bash
   cd vendor/searxng && python3 searx/version.py freeze
   ```
   (Also needed `git config --global --add safe.directory` for each
   submodule, since the checkout was rsynced in with different file
   ownership than the running user.)
3. **SearXNG disables JSON output by default.** `SearchProvider`'s adapter
   calls `?format=json`; upstream's default `settings.yml` only allows
   `html`. Fixed with `deploy/searxng/settings.yml` (mounted read-only),
   which extends the defaults and adds `json`.
4. **Crawl4AI refuses to bind beyond `127.0.0.1` without
   `CRAWL4AI_API_TOKEN`** (its own entrypoint's anti-open-proxy default) —
   this made it unreachable from `api`/`worker` over the *docker network*,
   not just from the host, since nothing (including sibling containers)
   can reach a loopback-only bind. Setting the token also turns on
   Bearer-auth enforcement, which `Crawl4AICrawlProvider` now sends on
   every request. First symptom: `httpx.ConnectError: All connection
   attempts failed` in worker logs.
5. **The worker crashed entirely on any DB error instead of logging and
   retrying.** `run_worker_loop`'s only `try/except` wrapped the crawl
   step; an error from `claim_next()` itself (first hit: querying
   `crawl_jobs` before migrations had run) propagated up and killed the
   whole process. No `restart:` policy existed yet either, so it stayed
   dead. Fixed: the entire loop iteration is now guarded, and every
   service has `restart: unless-stopped`.
6. **`mode: "semantic"` returned a raw unhandled 500**, not a clean error —
   `SemanticSearchNotImplementedError` had no registered FastAPI exception
   handler. Now returns `501 NOT_IMPLEMENTED` with a clear message.
7. **Polling `/v1/research` for a URL that hadn't resolved yet created a
   duplicate crawl job** on every call, instead of reusing the one already
   in flight — found by literally doing the documented background-mode
   polling pattern (ARCHITECTURE.md §5.2) against the live deployment.
   Fixed with `crawl_job_repository.get_active_by_url`, checked before
   every job creation.

## Known limitations (as of this deployment)

- **No git remote** — code was `rsync`'d to the server, not `git clone`d
  from anywhere. Redeploying a change means re-running the `rsync` +
  `docker compose build` + `up -d` sequence below; there's no `git pull`
  path on the server yet.
- **No admin endpoint for API keys** — the one key in use was inserted
  directly via `psql`. Fine for one trusted caller; revisit before adding
  more.
- **No rate limiting enforced** — `api_keys.rate_limit_per_minute` exists
  but nothing reads it yet (see `docs/nextsession.md`).
- **Redirect-hop SSRF revalidation gap** — unchanged from before this
  deployment; still tracked in `app/services/crawl/crawl4ai_provider.py`.
- **`docker system df` shows ~37GB of reclaimable image layers** from
  repeated rebuilds during setup — harmless (98G still free) but
  `docker image prune` is safe to run if space becomes a concern.

## Operating this deployment

**Redeploy a code change:**
```bash
# from the local checkout
rsync -az --exclude '.venv' --exclude '__pycache__' --exclude '.pytest_cache' \
  --exclude '.ruff_cache' --exclude '*.egg-info' --exclude 'dist' \
  --exclude '.env' --exclude 'vendor' --exclude '.git' \
  ./ root@ssr:/home/apps/deep-research-backend/

ssh root@ssr 'cd /home/apps/deep-research-backend && \
  docker compose build api worker && docker compose up -d'
```
Rebuild `searxng`/`crawl4ai` too only if their vendored source changed
(rare, slow — see BUILD.md §10.2).

**Apply a new migration:**
```bash
ssh root@ssr 'cd /home/apps/deep-research-backend && docker compose exec api alembic upgrade head'
```

**Check logs / health:**
```bash
ssh root@ssr 'cd /home/apps/deep-research-backend && docker compose ps -a'
ssh root@ssr 'cd /home/apps/deep-research-backend && docker compose logs <service> --tail 50'
```

**Scale workers:**
```bash
ssh root@ssr 'cd /home/apps/deep-research-backend && docker compose up -d --scale worker=3'
```

**Back up Postgres** (the only stateful service — `postgres_data` volume):
```bash
ssh root@ssr 'cd /home/apps/deep-research-backend && \
  docker compose exec postgres pg_dump -U postgres deep_research' > backup.sql
```
